## 2. 发现并修复 launcher JOB_TIMESTAMP race bug

### 症状

`fc_transip_strict_sharedrbf_paper_4M80ep` 这个 run 的 ckpt 散在两个目录:
- `..._20260518_052152/` — ranks 1-31 的 DCP shard
- `..._20260518_055240/` — rank 0 的 shard + `.metadata` + `resume.yaml`

任一目录单独都无法 DCP load (32 shards 必须在同一目录)。

### 根因

`src/predict_molecule/launchers/fairchem_torchrun.py:_broadcast_job_timestamp` 用 AFS 文件做跨 rank 时间戳同步: rank 0 写, 其他 rank poll-read。原配置 worker poll **60s 超时, 超时后静默落回 `datetime.now()`**。

SenseCore 这次的 pod 调度行为:
- worker pod 起在 05:21:52
- master pod 起在 05:52:40 (晚 30 min)

worker 60s 超时早就过了, 全部落回自己的 `datetime.now()` → ranks 1-31 用 `20260518_052152`, rank 0 用 `20260518_055240`, ckpt scatter。

### 修复

`src/predict_molecule/launchers/fairchem_torchrun.py:88-208`:
- Poll timeout 60s → 30 min (`POLL_ATTEMPTS=900 × 2s`), 覆盖 SenseCore 最坏 pod 调度延迟
- 静默 `datetime.now()` fallback → `raise RuntimeError` (失败要响, 别静默 scatter)
- 加 docstring 详记 2026-05-19 事件

Doc 写在 `ROADMAP.md` 死路 → 基础设施表 (新增一行)。
