- 速度那个是大工程,照 deepmd 的 make_fx 路子。torch.compile 直接上不行(明确不支持守恒力的二阶导),得先把带 .item()/.nonzero() 的几何(densify、PBC 邻居、min-image)全外提到 eager,力改成对预建的 edge_vec 求导再 scatter 回原子,中间那段纯 compute 才能被 make_fx 抓成扁平图给 inductor 融合。边通道的动态边数一开始卡住(make_fx 把边数 E 特化进 reshape),prime-padding 治不了,最后用"稀疏边 pad 到固定 E_max + mask 让 dummy 边失效"解了;stress 的 (I+ε) 应变、一图跨 batch 复用也都做通、E/力/应力全对 eager 到 1e-6。做成 backbone.use_fx_compile flag。
- 真训练 smoke 把隔离测试藏着的问题全挖出来了,最后诊断清楚、决定速度方案停在这。step 0 跑通(loss 5.9、grad_norm 35.8 非零 → 力反传确实穿过编译图给了参数梯度,机制是对的),中途还修了 DDP 要关 optimize_ddp、e3nn SH 直调纯多项式两个坑。但真配置 L=8 一直卡死,开 TORCH_LOGS 抓到根因:make_fx trace 其实很快(11.5 秒),真凶是 torch.compile/dynamo 每个 batch 都重编译 —— 它把"一个 batch 里的晶体数 B"这个每 batch 都变的维度锁死了,B 一变(87→92)就重编一次(每次 ~190 秒)、旧图还不释放 → 显存爬升卡死后 OOM。我只 pad 了边数 E,B/M/N 没管,漏了。改成 flat-replay(不经 inductor、直接跑符号图)后,能扛住变化的 B、不重编不 OOM,但对大模型 0.61x 比 eager 还慢 —— 摊平的 5200 节点图用 Python 逐算子跑,大图上比 eager 的 C++ autograd 引擎慢,玩具的 1.7x 不迁移。所以两个后端在真模型上都不给加速:inductor 要真加速得上 deepmd 整套动态形状机器(mark_dynamic 所有变化维)、工作量大回报未知,flat-replay 慢。速度方案就停这、flag 存档默认关。内存 flag(attn_chunk_size)是这轮真交付能用的。下一步回主线 scaling up。
